<objective>
Wire biomarkerService into the two screens that import BIOMARKERS directly: BiomarkerDetailScreen and BiomarkerEntryScreen.

Purpose: Screens no longer call the static BIOMARKERS import for their primary data source. They call getBiomarkers() on mount, which fetches from Supabase with static fallback. The Biomarker type shape is unchanged, so all downstream usage (rendering, navigation params, getStatus()) continues without modification.

Output: Both screens load biomarker definitions via the service layer.

**Scope decisions (intentional exemptions from D-02):**
- LongevityScoreScreen: does NOT import BIOMARKERS directly — confirmed by grep; out of scope.
- DashboardScreen: explicitly exempted by D-02 ("Dashboard can continue using the static array").
- GuidedFirstRunScreen: imports BIOMARKERS but uses it only to display 3 hardcoded biomarkers (Glucose, HbA1c, Cholesterol) in the guided onboarding flow. Static data is correct for this use case — the onboarding biomarker cards show fixed educational content, not ranges from the DB. Intentionally excluded from service migration.
