<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PaywallScreen → StoreKit | makePurchase() triggers Apple's native payment sheet — no card data touches app code |
| PaywallScreen → Adapty backend | Receipt validation happens server-side; Adapty returns isActive status |
| user input → purchase | No user text input in purchase flow — only button taps |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-06 | Tampering | Client-side purchase result | mitigate | makePurchase() result.type='success' triggers refreshPremium() which calls adapty.getProfile() — the server-side confirmed profile is the gate, not the local result flag alone. |
| T-16-07 | Information Disclosure | Product prices in UI | mitigate | All prices from product.price.localizedString (Adapty/StoreKit source). Never hardcoded — hardcoded prices cause App Store rejection AND may show wrong currency. |
| T-16-08 | Denial of Service | Adapty getPaywall() failure | accept | loadingProducts state prevents crash on empty product array. ActivityIndicator shown during load. Products returning empty array is handled gracefully (buttons show '...' price placeholder). Low risk — offline users cannot purchase anyway. |
| T-16-09 | Elevation of Privilege | IAP receipt spoofing | mitigate | Adapty performs server-side Apple receipt validation. The app only reads isPremium from adapty.getProfile() (server-confirmed). No client-side "I purchased successfully" flag persisted to AsyncStorage. |
</threat_model>
