## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| service → Supabase API | Write operations cross the network; user_id must come from server-validated auth.uid() via getUser(), never from client-provided data |
| StoredEntry → biomarker_entries row | biomarkerId field mapped to biomarker_id column; id preserved as-is; no other transformation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-04 | Spoofing | syncEntry user_id | mitigate | user_id is sourced from `supabase.auth.getUser()` — the server-issued anonymous session UUID — never from the StoredEntry payload or any caller-supplied value |
| T-08-05 | Tampering | biomarker_entries upsert | mitigate | RLS INSERT policy (T-08-01, Plan 01) enforces `user_id = auth.uid()` at the database level; even if client passes a wrong user_id, Supabase rejects the insert |
| T-08-06 | Denial of Service | fire-and-forget syncEntry | accept | syncEntry never blocks UI; network failure produces a console.warn and is silently dropped — no retry storm, no crash |
| T-08-07 | Information Disclosure | migrateHistory bulk upload | accept | Migration runs under the same anonymous session as all other Supabase calls; data is scoped to the device's auth.uid(); no cross-user data is accessible |
| T-08-SC | Tampering | npm/pip installs | accept | No new packages installed in this plan |
</threat_model>
