<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DashboardScreen → Supabase SELECT | Pull crosses network; user_id used as row filter; malformed rows would be mapped and could crash PhenoAge |
| App.tsx migration → AsyncStorage → Supabase | Raw JSON parsed from AsyncStorage before bulk insert; malformed JSON in @vitalspan_biomarkers would be caught by the .catch(() => null) chain |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-08 | Information Disclosure | DashboardScreen SELECT | mitigate | Query filters by `user_id` from `supabase.auth.getUser()` — the server-issued auth.uid(); RLS SELECT policy (Plan 01) provides a second enforcement layer |
| T-08-09 | Denial of Service | DashboardScreen sync pull | mitigate | Entire Supabase pull is wrapped in try/catch; on any error, setEntries falls back to AsyncStorage; UI never shows a blank screen or error state from a failed pull |
| T-08-10 | Tampering | Migration JSON parse | accept | `@vitalspan_biomarkers` is written only by the app itself; malformed JSON caught by `.catch(() => null)` in the migration chain; no silent data corruption possible |
| T-08-11 | Spoofing | Migration user_id | mitigate | migrateHistory calls `supabase.auth.getUser()` internally to get user_id; App.tsx never passes user_id — it is derived from the authenticated session inside the service |
| T-08-12 | Denial of Service | syncEntry per save() | accept | syncEntry is void / fire-and-forget; the catch inside the service absorbs all errors; BiomarkerEntryScreen.save() can never be blocked or throw from syncEntry |
| T-08-SC | Tampering | npm/pip installs | accept | No new packages installed in this plan |
</threat_model>
